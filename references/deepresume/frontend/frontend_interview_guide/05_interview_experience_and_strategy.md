## 公开面经的共性结论

### 结论一：前端一面仍然重基础

常见形式：

- JS 输出题：事件循环、闭包、`this`、原型链、Promise。
- CSS 布局：Flex/Grid、BFC、居中、响应式。
- 浏览器网络：缓存、跨域、渲染流程、Cookie/Token。
- 手写题：防抖、节流、深拷贝、Promise、并发控制。

你的策略：

- 不要只背概念，要能用输出题解释执行过程。
- 每个手写题写完都说边界条件和复杂度。
- 简历里出现的 SSE、WebSocket、`postMessage`、`iframe` 必须准备到项目级深度。

### 结论二：Vue 项目会被问“你只是调组件库吗？”

面试官会判断你是否真的理解组件状态和工程结构。常见追问：

- 组件如何拆分？
- 父子组件如何通信？
- Pinia/Vuex 里放什么状态？
- 路由权限如何设计？
- 图表组件如何销毁？
- 组件库二次封装如何控制耦合？

你的策略：

- 主动讲一个组件抽象案例，例如应用卡片、信息弹窗、图表组件、任务状态组件。
- 主动讲一个状态流，例如 AI 任务从 pending 到 running 到 success/error。
- 不要只说“使用 Ant Design Vue”，要说你封装了什么、暴露了什么 props/events、解决了什么复用问题。

### 结论三：AI 应用前端是你的差异化，但也容易被追问安全和异常

你的简历有 SSE、`iframe`、`postMessage`、DOM 注入、WebSocket，这些点很亮，但也很容易被深挖：

- SSE 断线如何处理？
- WebSocket 如何重连？
- `postMessage` 如何防止伪造消息？
- DOM 注入脚本是否有 XSS 风险？
- iframe 预览如何隔离不可信页面？
- 流式输出如何避免频繁渲染？

你的策略：

- 每个实时通信技术都准备“为什么选它、怎么实现、异常怎么处理、有什么安全边界”。
- 主动区分 SSE 和 WebSocket，不要混用概念。
- 主动说“前端权限不是安全边界，接口仍需后端校验”，这会显得成熟。

### 结论四：项目指标少时，要用可验证细节补强

当前简历没有大量量化指标，这不是致命问题，但面试讲述要更具体：

- 任务状态有哪些？
- 权限字段怎么设计？
- 组件 API 怎么设计？
- 消息协议有哪些类型？
- 出错时 UI 如何反馈？
- 如何清理定时器、WebSocket、图表实例和事件监听？

你的策略：

- 不编造性能提升百分比。
- 可以补“可测指标”：首屏加载时间、接口耗时、任务成功率、WebSocket 断线恢复率、图表渲染耗时、组件复用数量。
- 作品集或 GitHub README 里补流程图和截图，会显著提升可信度。

### 结论五：校招前端仍然需要算法题

常见形式：

- 笔试：2-4 道算法题，数组、字符串、哈希、栈、树、DP。
- 一面：1 道手写题 + 1 道算法题。
- 二面：算法减少，项目、工程化、浏览器增加。

你的策略：

- 前端刷题不追求偏难，优先中等题稳定 AC。
- 高频手写题必须默写无卡顿。
- 面试时先说思路和边界，再写代码。

## 面试流程打法

### 开场自我介绍

目标：让面试官知道你不是泛泛做前端，而是有 AI 应用型前端、实时通信和 B 端工程经验。

推荐结构：

```text
我目前是重庆邮电大学计算机技术硕士，求职方向是前端开发，比较关注 AI 应用型前端和 B 端工具类产品。
技术栈上主要使用 Vue3、TypeScript、Vue Router、Pinia、Ant Design Vue 和 ECharts。
项目方面，我做过云端智能图像协同系统，负责路由权限、AI 图像任务状态追踪、WebSocket 协同编辑和数据可视化；也做过智能应用生成平台，负责 SSE 流式对话、iframe 实时预览、postMessage 通信和可视化编辑。
另外我有一段 AI 应用研发实习，参与过模型工作流、接口设计和前后端联调，所以我比较希望应聘能结合前端工程、AI 交互和业务系统交付的岗位。
```

注意：

- 第一句明确前端方向。
- 第二句抛出技术栈。
- 第三句只讲最强项目，不要把所有经历平铺。
- 控制在 1 分钟以内，给面试官留下追问入口。

### 项目讲述打法

使用“P-T-E-R-B”结构：

- Problem：业务问题和技术难点。
- Task：你负责什么。
- Engineering：具体方案和技术选择。
- Result：结果、体验、工程价值。
- Bad Case：失败案例和迭代。

示例：智能应用生成平台

```text
这个项目的问题是 AI 生成网站不能只返回一段代码，用户需要边对话边看到实时预览，并且能直接点击页面元素修改。
我负责前端交互链路，主要做了三块：第一是用 EventSource 消费 SSE 流式接口，实现 AI 对话的增量输出和工具调用轨迹展示；第二是用 iframe 承载预览页面，通过 postMessage 和主应用同步选中元素、编辑内容和预览状态；第三是向预览页注入 DOM 监听脚本，实现点击即修改。
异常处理上，我维护了 streaming、done、error 等状态，切换会话时会关闭旧连接；postMessage 通信会校验消息类型和来源，避免错误消息污染状态。
如果重做，我会把消息协议做得更规范，增加 requestId、错误码和更严格的 sandbox 策略。
```

### 回答追问打法

遇到追问时按这个顺序：

1. 先给直接结论。
2. 再说明原因或原理。
3. 再联系项目实践。
4. 最后补充局限和改进。

示例：

```text
SSE 更适合这个场景，因为 AI 对话主要是服务端持续向浏览器推送 token，客户端不需要高频双向通信。
它基于 HTTP，浏览器原生提供 EventSource，前端实现和调试都比较轻。
在我的项目里，我用它消费后端流式接口，收到消息后增量更新对话内容，并根据结束事件切换状态。
它的局限是只能单向推送，如果后续要做多人协同编辑或复杂双向控制，我会选择 WebSocket。
```

### 不会的问题怎么处理

可以这样说：

```text
这个点我没有在线上项目里完整实践过，但我理解它的基本思路是……
如果让我在项目中落地，我会先从……开始验证。
```

不要这样说：

- “这个我没看过。”
- “组件库已经做好了。”
- “后端会处理。”
- “网上一般都是这么写的。”

## 一面、二面、HR 面准备重点

### 一面

重点：

- JS/CSS/浏览器基础。
- Vue 基础。
- 手写题和算法题。
- 简历项目初步追问。

准备：

- 每天 2-3 道手写/算法题。
- JS 输出题和事件循环背熟并能解释。
- 自我介绍控制在 1 分钟。

### 二面

重点：

- 项目深挖。
- 工程化、性能、安全。
- 技术取舍和异常处理。
- 与后端、产品、测试协作。

准备：

- 每个项目画流程图。
- 每个项目准备 3 个 Bad Case。
- 每个实时通信方案准备替代方案。
- 准备“如果重做会怎么优化”。

### HR 面

重点：

- 求职动机。
- 稳定性。
- 沟通协作。
- 对岗位理解。
- 实习时间和毕业安排。

准备：

- 为什么选择前端开发。
- 为什么强调 AI 应用型前端。
- 研究生期间如何安排实习和毕业。
- 项目中如何和后端、算法、产品协作。

## 反问问题库

技术面可以问：

- 团队前端技术栈主要是 Vue 还是 React？是否有统一组件库或工程化基建？
- 这个岗位更偏业务页面、B 端平台、可视化，还是 AI 应用交互？
- 团队如何做前端质量保障，例如 Code Review、单测、E2E、监控和性能指标？
- 如果涉及 AI 应用，前端通常如何和模型服务、流式接口或 Agent 工作流协作？

HR 面可以问：

- 这个岗位对应届生或实习生的培养路径是怎样的？
- 实习转正或校招流程的时间安排大概是怎样的？
- 团队对前端工程能力和业务理解能力的权重如何？

避免反问：

- 能不能不加班？
- 多久涨薪？
- 我这个表现能过吗？
- 你们前端是不是只写页面？

## 投递策略

优先投递 JD 中出现这些关键词的岗位：

- Vue3、TypeScript、Vite、Pinia、Vue Router
- B 端系统、管理后台、组件库、前端工程化
- ECharts、数据可视化、报表、看板
- WebSocket、SSE、实时通信、协同编辑
- 低代码、在线编辑器、可视化搭建、AI 应用平台
- 前后端联调、接口设计、性能优化

简历和投递材料建议：

- GitHub README 补充项目截图、流程图、技术架构图和关键交互 GIF。
- 为智能应用生成平台准备一个可访问 Demo 或录屏，AI 前端体验靠文字很难完全表达。
- 将简历文件命名为 `前端开发-无绝-实习.pdf` 或按公司要求命名。
- 每投一个 JD，按关键词微调技能顺序：B 端强调组件/表格/权限，AI 前端强调 SSE/iframe/WebSocket，数据可视化强调 ECharts/性能。

如果投 React 岗：

- 简历可以投，但要补 React Hooks、状态管理、React Router、组件性能优化。
- 面试中不要把 Vue 经验包装成 React 熟练，稳妥说法是“主栈 Vue3，但组件化、状态管理、工程化和浏览器基础可迁移，React 正在补齐”。
