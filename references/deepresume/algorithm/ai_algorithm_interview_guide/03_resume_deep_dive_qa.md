# 简历项目深挖模拟 QA

## 使用方式

每个项目至少准备三层回答：

- 30 秒：一句话说清项目价值。
- 3 分钟：背景、职责、方案、结果。
- 15 分钟：能展开架构、指标、细节、失败案例和优化方向。

回答原则：

- 不要只说“用了 Dify / LangChain / vLLM”，要说你为什么用、怎么接、遇到什么问题、怎么评估。
- 指标必须能解释口径，例如 Hit Rate、执行成功率、业务采纳率、Golden Dataset。
- 遇到不确定数据，用“当时我们的口径是……”而不是编造。

## 项目一：基于 RAG 的医药数据分析问答助手

### 项目 30 秒版本

这个项目面向药品采购、零售、库存等业务数据，底层有 800 多张业务表。我的主要工作是设计 RAG + Text-to-SQL 链路，让销售人员能用自然语言提问，系统自动召回相关表字段、生成 SQL、执行校验并输出分析结论。项目通过 Schema 检索、SQL 预执行自修正和 Golden Dataset 评估，把 SQL 执行成功率从约 70% 提升到 90%+。

### Q1：这个项目的核心难点是什么？

**面试官可能问：** 这个项目不就是自然语言转 SQL 吗？难点在哪里？

**建议回答：**  
难点主要有三个。第一是 Schema 规模大，有 800 多张业务表，用户问题里不会直接出现标准表名字段名，所以需要先做 Schema Linking。第二是业务字段存在别名、枚举值和隐含规则，例如销售额、库存、采购量可能分布在不同表里，单靠表名相似度不够。第三是 SQL 生成后不能只看语法正确，还要能执行、结果口径合理，并且要有错误回传和 Bad Case 回流。

**可能追问：** 你怎么做 Schema Linking？

**追问回答方向：**  
我把表名、字段名、字段中文含义、常见枚举值、业务备注拼接成富文本后向量化；查询侧先做问题类型判断和关键词抽取，再召回候选表字段；最后结合业务规则过滤和样例 SQL 约束，让模型优先使用高相关表。对于强关键词字段，例如药品编码、门店、供应商，会保留关键词匹配结果，避免纯语义召回漏掉精确字段。

### Q2：为什么要做元数据增强？

**面试官可能问：** 表名和字段名直接向量化不行吗？

**建议回答：**  
直接向量化表名和字段名效果不稳定，因为很多企业表名是缩写或内部命名，例如 `inv_sku_flow` 这类名字对模型和 embedding 都不友好。元数据增强的作用是把机器字段转成业务语义，把字段中文含义、枚举值、单位、指标说明拼进去后，用户问题和字段描述之间的语义距离更近。这样可以提升候选表字段召回质量，也能减少模型生成 SQL 时选错字段。

**可能追问：** 元数据太长会不会影响召回？

**追问回答方向：**  
会，所以不能无脑拼接。实践中可以把字段说明控制在固定模板里，例如“表用途、主键、关键字段、枚举值、关联表”；过长的表说明可以拆成表级 chunk 和字段级 chunk；召回后再按字段类型、业务域、关键词命中进行过滤。

### Q3：800 张表如何避免召回过多导致模型上下文爆炸？

**建议回答：**  
我的思路是分层召回。第一层按业务域或问题类型粗筛，例如采购、销售、库存、门店；第二层做表级召回，选出少量候选表；第三层在候选表内做字段级召回；最后只把必要的表结构、字段含义、关联关系和样例 SQL 放入 Prompt。这样比一次性把大量 Schema 塞给模型更稳定。

**可能追问：** 如果问题需要跨域 Join 怎么办？

**追问回答方向：**  
可以维护表关系图或常见 Join 路径。召回时除了相关表，还要扩展一跳或两跳关联表，例如事实表关联药品维表、门店维表、时间维表。生成前把可用 Join key 明确写入上下文，并限制模型使用这些 Join 条件。

### Q4：SQL 自纠错是怎么做的？

**建议回答：**  
生成 SQL 后先走预执行或 Explain，捕获 MySQL 返回的语法错误、字段不存在、表不存在、聚合错误等信息。然后把错误类型、错误 SQL、相关 Schema 和用户原问题一起回传给 LLM，让模型重写 SQL。为了避免无限循环，需要限制重试次数，并记录每次修改差异；如果连续失败就降级到人工反馈或返回无法确定。

**可能追问：** 预执行会不会有安全风险？

**追问回答方向：**  
会，所以要用只读账号、SQL 类型白名单，只允许 SELECT；限制访问表范围、超时时间和返回行数；对敏感字段做脱敏；禁止 UPDATE、DELETE、INSERT、DROP 等语句；必要时先解析 SQL AST，而不是直接执行模型输出。

### Q5：执行成功率从 70% 到 90%+ 是怎么定义的？

**建议回答：**  
我会把指标拆开讲。执行成功率主要指 SQL 能够通过语法和数据库执行，不报错；它不完全等价于答案正确。为了验证结果是否可靠，还需要在 Golden Dataset 上看结果正确率或人工验收通过率。70% 到 90%+ 是在约 200 条 Golden Dataset 上统计的执行成功口径，提升来自 Schema 检索优化、样例约束和 SQL 自纠错。

**可能追问：** 如果 SQL 执行成功但结果错了怎么办？

**追问回答方向：**  
这类问题要靠结果级评估和人工 Bad Case。可以做结果一致性检查，例如对标准答案 SQL 的结果做对比；对聚合口径、时间范围、过滤条件做规则校验；同时把错误样本回流到样例库和 Schema 描述中。

### Q6：RAGAS 在 Text-to-SQL 中怎么用？

**建议回答：**  
RAGAS 更适合评估 RAG 问答中的上下文相关性、忠实度、答案相关性等。Text-to-SQL 场景不能只依赖 RAGAS，因为最终目标是 SQL 可执行且结果正确。我更倾向于把 RAGAS 用在 Schema/上下文召回质量评估上，再结合 SQL 执行准确率、语法正确率、结果一致性和人工反馈做完整评估。

**可能追问：** 哪个指标最重要？

**追问回答方向：**  
离线阶段看候选 Schema Recall@K 和执行准确率；上线前看结果正确率和人工验收；业务阶段看用户采纳率、纠错率和人工介入率。不同阶段指标不同，不能只看一个分数。

### Q7：为什么用 Dify 和 LangChain？

**建议回答：**  
Dify 适合快速搭建可视化工作流、模型接入、Prompt 编排和人工反馈闭环；LangChain 更适合在代码侧封装检索、工具调用、链路组合和自定义逻辑。项目中如果只是简单 demo，Dify 足够；但涉及 Schema 检索、自定义 Rerank、SQL 自纠错和服务集成时，需要 LangChain 或自写代码补足灵活性。

**可能追问：** 如果不用 Dify 你能实现吗？

**追问回答方向：**  
可以。核心链路是数据预处理、向量索引、查询理解、Schema 召回、Prompt 构造、SQL 生成、预执行、自修正、结果解释和反馈记录。Dify 是编排和管理工具，不是不可替代的核心算法。

## 项目二：小啄 GPT 智能回复模型

### 项目 30 秒版本

这个项目面向家电售后客服多轮对话，问题是模型容易丢上下文、漏槽位、知识检索不准。我设计了短期记忆、长期摘要和结构化状态机制，用 JSON 维护家电类型、故障现象、预约时间等关键槽位；同时对 Qwen 做领域微调，并用 Dense/BM25、RRF、Rerank 优化知识检索，使业务采纳率提升约 10%，长尾 Query Hit Rate 提升约 20%。

### Q1：为什么需要“短期记忆 + 长期摘要 + 结构化状态”？

**建议回答：**  
客服对话有两个特点：轮次长、状态强。短期记忆保留最近几轮原文，避免丢失用户刚说的信息；长期摘要压缩历史对话，避免上下文过长；结构化状态用于维护关键槽位，例如家电类型、故障现象、地址、预约时间。三者结合比单纯把历史对话塞进 Prompt 更稳定，也方便后续工具调用和流程控制。

**可能追问：** JSON 状态如何更新？

**追问回答方向：**  
每轮对话后让模型或规则抽取槽位，只更新高置信字段；新信息和旧信息冲突时保留冲突标记或要求用户确认；关键字段可以设置必填、可选、待确认状态；输出前根据 JSON 状态判断是否能进入下一流程。

### Q2：怎么避免模型乱改槽位？

**建议回答：**  
可以从三层控制。第一是 Prompt 明确只允许更新用户本轮明确表达的信息；第二是用 JSON Schema 或 Pydantic 校验字段类型和值域；第三是对关键字段设置置信度和确认机制，例如预约时间、地址、手机号不能仅凭模型推断。

**可能追问：** 如果用户前后说法冲突怎么办？

**追问回答方向：**  
不能直接覆盖。应该记录候选值和来源轮次，让模型向用户澄清，例如“您刚才说是周三，之前说是周五，请确认哪个时间方便”。这比静默覆盖更符合客服场景。

### Q3：为什么要微调 Qwen，而不是只靠 Prompt？

**建议回答：**  
Prompt 能快速控制格式和流程，但对于家电维修领域的标准话术、常见故障表达和回复风格，如果仅靠 Prompt，稳定性和一致性不够。微调 Qwen 的目标不是把所有知识塞进模型，而是让模型更适应客服话术、领域表达和结构化输出习惯。事实性知识仍然主要依赖 RAG。

**可能追问：** 微调数据怎么构造？

**追问回答方向：**  
数据可以来自历史客服对话、标准话术、维修知识库和人工标注样本。需要清洗隐私信息、去重、过滤低质量回复，并统一成指令格式。训练/验证集要按问题类型和产品类型分层，避免同类样本泄漏。

### Q4：LoRA 参数怎么选？

**建议回答：**  
LoRA 的 rank 影响可训练参数量和表达能力，rank 越大越能拟合任务但越容易过拟合、显存开销也更高；alpha 控制低秩更新的缩放；target modules 一般选择 attention 或 MLP 中对任务影响较大的线性层。实际选择要看显存、数据量、验证集效果和生成稳定性。

**可能追问：** 如何判断微调是否过拟合？

**追问回答方向：**  
看训练/验证 loss 差距、验证集任务指标、人工抽检质量，以及模型是否过度套用训练话术。客服场景还要看幻觉率、槽位准确率和拒答/澄清能力。

### Q5：Dense/BM25/RRF/Rerank 为什么这样组合？

**建议回答：**  
BM25 对关键词、型号、故障码、品牌名这类精确匹配更敏感；Dense 检索能处理口语化表达和语义相似问题。两路召回分数尺度不同，直接加权不稳定，所以用 RRF 按排名融合。融合后候选数量仍较多，再用 Rerank 做精排，提升 TopK 结果相关性。

**可能追问：** Rerank 会增加延迟，怎么权衡？

**追问回答方向：**  
可以限制 Rerank 候选数量，例如只对前 20 或 50 个候选重排；对高置信关键词命中问题直接返回；对低风险问题降级不用 Rerank；线上根据 TP95 延迟和回答质量做权衡。

### Q6：Hit Rate 提升 20% 怎么统计？

**建议回答：**  
Hit Rate 通常是在标注查询集上统计：如果 TopK 检索结果中包含人工标注的相关文档，就算命中。需要说明 K 值、测试集规模、查询类型和 baseline。如果说提升 20%，最好说明是相对提升还是绝对百分点提升。

**可能追问：** Hit Rate 提升一定会带来回答质量提升吗？

**追问回答方向：**  
不一定。检索命中只是前提，最终回答还受上下文拼接、Prompt、模型生成、引用约束影响。所以还要结合人工评分、幻觉率、业务采纳率等指标。

### Q7：业务采纳率提升 10% 怎么定义？

**建议回答：**  
业务采纳率可以定义为客服或业务人员直接采用模型回复的比例。这个指标要说明统计周期、样本量、是否排除无效对话、是否和旧版本 A/B 对比。它比离线指标更接近业务价值，但也更容易受流量、人工习惯和问题分布影响。

**可能追问：** 如果面试官质疑这个指标怎么办？

**追问回答方向：**  
可以承认业务指标有噪声，所以我会同时看离线检索指标、人工标注质量、幻觉率和线上采纳率。10% 是业务统计口径下的提升，不把它包装成纯算法指标。

## 项目三：多模态家电识别与知识库入库系统

### 项目 30 秒版本

这个项目用于家电售后知识库建设，目标是从图片、手册或产品信息中识别品牌、型号、规格、品类等字段，并自动补全、校验、去重、入库。我负责设计“识别--补全--校验--去重--入库”流程，结合 OCR/视觉大模型、联网搜索、字段置信度校验、向量相似度和规则去重，并用 FastAPI 封装成可复用服务。

### Q1：OCR 和视觉大模型如何分工？

**建议回答：**  
OCR 更适合提取图片中的明确文字，例如型号、铭牌、规格参数；视觉大模型更适合理解图片整体内容，例如判断家电品类、识别外观特征、补充 OCR 漏掉的信息。实际流程中可以先 OCR 提取可验证文本，再让视觉大模型结合图片和 OCR 结果做结构化字段抽取。

**可能追问：** OCR 错了怎么办？

**追问回答方向：**  
可以用规则校验型号格式、品牌词典、联网搜索结果和知识库已有记录做交叉验证。低置信字段不直接入库，而是进入二次补全或人工审核。

### Q2：联网搜索如何控制幻觉？

**建议回答：**  
联网搜索只能作为补全候选，不能直接作为最终事实。需要记录来源、摘要、字段置信度；多个来源一致时提高置信度；和 OCR 结果冲突时要求人工确认或标记低置信。Prompt 中也要要求模型只基于搜索结果抽取字段，不允许自由编造。

**可能追问：** 如果不同网页信息冲突怎么办？

**追问回答方向：**  
优先使用官网、说明书、可靠电商详情页等可信来源；记录冲突字段；如果关键字段冲突，不能自动入库，应进入待确认状态。

### Q3：字段置信度如何设计？

**建议回答：**  
可以综合来源类型、OCR 置信度、规则校验、搜索一致性、知识库相似记录匹配结果。例如型号字段如果 OCR 清晰命中、符合型号正则、搜索结果一致，就给高置信；如果只由模型推断且无外部证据，就给低置信。

**可能追问：** 置信度阈值怎么定？

**追问回答方向：**  
先用标注样本调阈值，看 Precision/Recall；对于入库这种低容错场景，更偏向高 Precision，宁愿人工确认也不要污染知识库。

### Q4：如何做重复入库判断？

**建议回答：**  
重复判断不能只看向量相似度。我的思路是把识别文本、结构化字段和知识库摘要向量化，先召回相似候选，再用规则校验品牌、型号、规格等关键字段。如果型号完全一致且品牌一致，基本可以判重；如果型号缺失但描述相似，就需要结合置信度和人工确认。

**可能追问：** 向量相似度阈值怎么选？

**追问回答方向：**  
用历史样本构造正负样本对，画相似度分布，选择能兼顾误判和漏判的阈值。入库场景更怕误合并，所以阈值通常要保守，并叠加规则。

### Q5：为什么要接 vLLM？

**建议回答：**  
vLLM 主要用于提高大模型推理服务的吞吐和并发能力。多模态入库流程中可能有批量图片或手册处理，请求长度和生成长度不一，普通逐请求推理 GPU 利用率不高。vLLM 的连续批处理和 KV Cache 管理可以提升服务效率。

**可能追问：** 接入 vLLM 后如何评估？

**追问回答方向：**  
看 QPS、平均延迟、TP95/TP99、GPU 利用率、显存占用、错误率。也要区分首 token 延迟和整体生成耗时。

### Q6：FastAPI 服务如何设计接口？

**建议回答：**  
可以拆成识别、补全、去重、入库几个接口，也可以提供一个端到端接口。工程上要有请求 ID、任务状态、错误码、日志、超时、文件大小限制和鉴权。耗时任务可以异步化，前端轮询任务状态或使用消息队列。

**可能追问：** 如果接口超时怎么办？

**追问回答方向：**  
长任务异步化；设置合理超时；中间结果落库；失败可重试；对模型服务设置熔断和降级策略，例如只返回 OCR 结果或进入人工处理。

## 论文：《LHAF: A Lightweight Hierarchical Adaptive Framework for LLM-based Multimodal Emotion Recognition in Conversations》

### Q1：这篇论文解决什么问题？

**建议回答：**  
这篇论文面向多模态对话情绪识别，目标是在对话上下文中结合文本、视觉或音频等模态信息，提升情绪识别效果，同时保持框架轻量。面试中我会重点讲问题定义、模型结构、为什么需要层次化和自适应机制、实验指标和消融结果。

**可能追问：** 和你的求职方向有什么关系？

**追问回答方向：**  
它证明我有多模态建模和论文实验能力；和 AI 应用开发结合点在于多模态理解、对话建模和模型评估，这些能力可以迁移到多模态 Agent 或客服对话分析中。

### Q2：在投论文怎么讲才稳妥？

**建议回答：**  
可以说“SCI I 区在投”，但不要把它当成已录用成果。重点讲自己实际负责的部分，例如模型设计、实验、数据处理、消融分析或论文撰写。避免夸大贡献。

**可能追问：** 如果问到实验结果？

**追问回答方向：**  
只说你能确认的数据。如果结果仍在投稿保密或未最终确定，可以说“具体数值以投稿版本为准，但我可以讲实验设计和主要结论”。

## 通用追问库

### 技术选择

**问题：** 为什么选择这个方案，而不是另一个方案？  
**回答框架：** 业务约束、数据特点、成本、延迟、可维护性、评估结果。

### 指标可信度

**问题：** 你的提升指标怎么来的？  
**回答框架：** 测试集规模、统计口径、baseline、对比方法、样本周期、局限性。

### Bad Case

**问题：** 失败案例有哪些？  
**回答框架：** 分类列举，例如召回错、生成错、数据脏、权限问题、模型幻觉；再说如何修复。

### 工程落地

**问题：** 项目上线了吗？如何部署？  
**回答框架：** 服务架构、接口、容器、日志、压测、错误处理、人工反馈。

### 个人贡献

**问题：** 哪些是你做的？  
**回答框架：** 我负责的模块、关键决策、实现细节、结果验证、协作边界。
