# 简历项目深挖模拟 QA

## 使用方式

每个项目至少准备三层回答：

- 30 秒：一句话说清项目价值。
- 3 分钟：背景、职责、方案、结果。
- 15 分钟：能展开架构、数据、断言、报告、失败案例和优化方向。

回答原则：

- 不要只说“我用了 Pytest / Playwright / JMeter”，要说为什么用、怎么封装、怎么验证、如何定位失败。
- 指标必须能解释口径，例如 100+ 对话回归样例、QPS、P95、错误率、通过率。
- 没有证据的数据不要编造。可以说“这个指标我当时没有量化，但我会这样设计统计口径”。

## 项目一：重庆啄木鸟网络科技有限公司实习

### 项目 30 秒版本

这段实习主要是 AI 应用测试与交付验证。我参与大模型应用研发和交付，负责模型服务接口联调、Prompt/RAG 流程测试、冒烟与回归用例设计、接口异常排查和线上 Bad Case 分析。具体做过小啡 GPT 智能回复模型和智能家电识别入库系统，重点是把模型输出变成可验证、可回归、可定位的问题。

### Q1：你实习的核心职责是什么？

**面试官可能问：** 算法实习生为什么做测试？你具体负责哪些事情？

**建议回答：**  
我的职责更偏 AI 应用测试与交付验证，不是纯模型训练。我主要做三类事情：第一是模型服务接口联调，保证上下游接口、请求参数、响应格式和异常处理可用；第二是 Prompt/RAG 流程测试，设计冒烟、回归和异常场景；第三是线上 Bad Case 分析，把上下文丢失、知识召回错误、格式漂移等问题沉淀成回归样例。

**可能追问：** 你和普通功能测试有什么区别？

**追问回答方向：**  
普通功能测试更多是确定性输入输出，AI 应用测试面对的是不确定输出，所以不能只做精确字符串断言。我会结合关键词、JSON Schema、相似度、规则断言和人工标注样例，还要关注 Prompt 版本、检索结果、模型参数和上下文状态。

### Q2：Prompt/RAG 流程怎么测试？

**建议回答：**  
我会把链路拆开看：请求参数是否正确、检索是否命中、上下文拼接是否包含必要知识、模型输出是否符合业务规则和格式要求。测试集上分为正常问法、口语化问法、多轮上下文、缺少信息、错误信息、越界问题和拒答场景。断言上对结构化输出做 JSON Schema 校验，对知识命中做关键词或相似度校验，对高风险问题做人工验收。

**可能追问：** RAG 没有标准答案怎么办？

**追问回答方向：**  
可以先构建 Golden Dataset，为核心问题标注期望知识片段或验收规则。对于没有唯一答案的问题，用多维度评估，例如是否引用正确知识、是否拒绝编造、是否输出必要字段、是否符合客服话术。自动化断言覆盖稳定规则，主观质量用人工抽检补充。

### Q3：100+ 条对话回归样例怎么沉淀？

**建议回答：**  
来源主要是测试和线上 Bad Case。每条样例会记录用户多轮输入、期望槽位状态、应命中的知识点、期望回复约束和实际失败类型。失败类型可以分为上下文丢失、知识召回错误、格式漂移、指令不遵循、重复追问、拒答不当等。后续模型、Prompt 或知识库更新后，用这些样例做回归对比。

**可能追问：** 如何避免回归样例越来越乱？

**追问回答方向：**  
需要分类和版本管理。按业务场景、问题类型、失败原因、优先级分组；同时记录样例来源和最后验证版本。无效样例要归档，核心主链路样例要稳定保留。

## 项目二：小啡 GPT 智能回复模型

### 项目 30 秒版本

这个项目面向家电客服多轮对话，我围绕意图识别、槽位抽取、知识命中、指令遵循和结构化输出构建测试集。通过脚本批量调用模型 API，对回复内容做关键词、JSON Schema、相似度和规则断言，沉淀 100+ 条对话回归样例，定位上下文丢失、知识召回错误和格式漂移等问题。

### Q1：多轮对话测试重点是什么？

**建议回答：**  
重点是状态是否持续正确。比如用户第一轮说空调不制冷，第二轮补充品牌型号，第三轮说想预约维修，系统要保留家电类型、故障现象、品牌型号、预约意图等槽位。测试时不仅看单轮回复，还要看历史信息有没有丢、冲突信息是否澄清、关键字段是否被正确更新。

**可能追问：** 如何设计测试数据？

**追问回答方向：**  
按场景分层：完整信息、信息缺失、用户纠正、前后冲突、口语化表达、多意图混合、无关问题、敏感问题。每类准备若干轮对话，并标注期望槽位和回复约束。

### Q2：JSON Schema 校验有什么用？

**建议回答：**  
如果模型需要输出结构化结果，例如意图、槽位、回复类型和下一步动作，JSON Schema 可以校验字段是否存在、类型是否正确、枚举值是否合法。这能发现格式漂移问题，比如少字段、字段类型错误、输出自然语言混在 JSON 里。

**可能追问：** Schema 通过就说明回复正确吗？

**追问回答方向：**  
不说明。Schema 只保证格式和基础字段合法，内容正确还要看知识命中、业务规则、槽位准确性和人工验收。所以我会把 Schema 校验作为第一层，后面叠加关键词、相似度和规则断言。

### Q3：相似度断言怎么用，风险是什么？

**建议回答：**  
相似度断言适合判断回复是否接近期望表达或是否命中某类语义，但不能单独作为最终判定。风险是相似度高不代表事实正确，相似度低也可能是合理改写。所以更适合做辅助指标，结合关键字段和人工规则一起判断。

**可能追问：** 如何降低误判？

**追问回答方向：**  
可以设置不同场景的阈值，对高风险问题使用更严格的规则和人工复核；对结构化字段优先用精确断言；对开放性回复只判断必要信息是否出现。

## 项目三：智能家电识别与自动入库系统

### 项目 30 秒版本

这个系统针对 OCR/视觉大模型识别、联网补全、向量去重和入库链路做接口测试与数据质量校验。我用 Postman/Python 构造正常、缺字段、重复商品、低置信度识别等场景，通过数据库断言验证字段完整性、去重结果和入库状态，提升交付前缺陷发现效率。

### Q1：这个系统的测试难点是什么？

**建议回答：**  
难点在于输入不稳定、链路长、数据质量要求高。图片识别可能出现 OCR 错字、字段缺失、置信度低；联网补全可能信息冲突；向量去重可能误合并或漏判；入库还要保证字段完整、状态正确和可追溯。所以测试不只看接口是否成功，还要验证数据质量和入库副作用。

**可能追问：** 如何构造测试场景？

**追问回答方向：**  
我会构造正常图片、缺少型号、品牌模糊、重复商品、低置信度、联网信息冲突、非法图片格式、超大文件、重复提交等场景。每个场景都要校验响应、数据库字段、去重状态和错误日志。

### Q2：数据库断言怎么设计？

**建议回答：**  
接口返回成功后，需要查询对应入库记录，验证品牌、型号、品类、规格、置信度、来源、去重状态和审核状态。对于重复商品，要验证不会产生脏数据；对于低置信字段，要验证是否进入待确认状态，而不是直接入库。

**可能追问：** 自动化测试直接查数据库会不会不好？

**追问回答方向：**  
需要控制边界。测试环境可以用只读查询验证接口副作用，但不要随意改生产数据。自动化中应使用独立测试数据、清理脚本和环境隔离。

## 项目四：接口自动化测试框架

### 项目 30 秒版本

这个项目是面向登录、用户、权限、商品、订单等接口搭建可复用自动化框架。我按配置层、API 客户端层、业务关键字层、用例层和报告层拆分，封装环境切换、Token 自动刷新、统一请求日志、公共断言、数据库校验、Fixture 和参数化数据驱动，并接入 Allure/JUnit XML 和 CI。

### Q1：为什么要做框架分层？

**建议回答：**  
分层是为了降低维护成本。配置层管理环境和账号；API 客户端层统一请求、Token、日志和异常；业务关键字层封装登录、下单等复用动作；用例层只关注测试场景和断言；报告层负责结果展示。这样接口变更时只需要改客户端或关键字层，不必改大量用例。

**可能追问：** 如果接口字段变了怎么办？

**追问回答方向：**  
如果是请求/响应字段变化，优先修改客户端封装和 Schema；如果影响业务动作，修改关键字层；最后再更新对应用例数据。框架要能通过报告快速定位是哪个接口、哪个字段、哪个断言失败。

### Q2：Token 自动刷新怎么做？

**建议回答：**  
可以在客户端层维护登录态。请求前检查 Token 是否存在或即将过期；如果响应出现 401 或特定错误码，再触发重新登录并重试一次。为避免并发用例重复刷新，可以在 session 级 Fixture 中初始化 Token，必要时加锁或统一刷新入口。

**可能追问：** 重试会不会掩盖问题？

**追问回答方向：**  
会，所以要限制重试次数并记录日志。只有明确是 Token 过期或临时网络错误才重试；业务断言失败不能重试掩盖。

### Q3：契约测试和异常测试怎么做？

**建议回答：**  
契约测试基于 OpenAPI/Swagger 文档，验证状态码、字段类型、必填字段和响应 Schema。异常测试会构造边界值、非法枚举、缺少必填字段、嵌套对象异常等数据，验证接口是否返回正确错误码和提示，同时输出失败接口的请求参数、响应体和可复现 curl。

**可能追问：** OpenAPI 文档不准怎么办？

**追问回答方向：**  
这本身就是质量问题。可以把文档和实际响应差异记录为契约不一致缺陷，同时和后端确认以代码为准还是文档为准。契约测试的价值之一就是推动接口文档和实现同步。

### Q4：CI 中跑哪些用例？

**建议回答：**  
不是所有用例都适合每次提交跑。代码提交后优先跑冒烟集合和核心接口；定时任务跑完整回归；性能和稳定性测试单独触发。用 mark 区分 smoke、regression、slow、module 标签，CI 根据场景选择执行。

**可能追问：** CI 用例失败怎么处理？

**追问回答方向：**  
先看是环境问题、数据问题、脚本问题还是产品缺陷。报告要保留请求响应、日志、数据库校验结果和历史失败。不能一失败就重跑到绿，要先确认失败原因。

## 项目五：持续测试平台与性能压测

### 项目 30 秒版本

这个项目参考 MeterSphere、Playwright 和 Locust，使用 Vue.js、FastAPI、MySQL、Redis、Docker 搭建轻量测试平台，支持项目管理、用例维护、测试计划、任务记录、缺陷备注和报告归档。平台把 Pytest、Playwright 和 Locust 执行入口封装为任务，通过 Redis 队列异步调度并回收结果。

### Q1：为什么要做测试平台，而不是直接命令行跑脚本？

**建议回答：**  
命令行适合个人执行，但团队需要统一管理用例、计划、任务、报告和历史趋势。测试平台可以让不同成员提交任务、查看结果、归档报告、跟踪失败原因，并把接口、UI 和性能测试结果统一起来。平台化的价值是降低执行门槛和提升质量数据可见性。

**可能追问：** 平台最小可用版本需要哪些功能？

**追问回答方向：**  
项目管理、用例管理、任务触发、任务状态、报告归档、失败日志、历史记录。高级功能如权限、趋势分析、环境管理可以后续迭代。

### Q2：Redis 队列如何调度任务？

**建议回答：**  
平台提交任务后，把任务 ID、类型、参数、环境、创建人写入数据库，同时将任务 ID 推入 Redis 队列。Worker 从队列取任务，根据类型调用 Pytest、Playwright 或 Locust 执行，实时更新任务状态，结束后保存报告路径、通过率、失败原因和耗时。

**可能追问：** 任务失败或 Worker 挂了怎么办？

**追问回答方向：**  
需要任务状态机和超时机制。任务可以有 pending、running、success、failed、timeout 状态；Worker 执行时写心跳；超时任务可标记失败或重新入队；日志和中间结果要落库，避免完全丢失。

### Q3：平台如何支持性能测试？

**建议回答：**  
平台把 Locust/JMeter 的执行参数配置化，例如并发数、加压时间、持续时间、目标地址和数据文件。执行后聚合 QPS、响应时间、P95、错误率和慢接口，并把报告归档。性能测试最好单独环境执行，避免影响普通自动化任务。

**可能追问：** 性能瓶颈怎么定位？

**追问回答方向：**  
先看压测报告是否有错误率和慢接口，再结合服务端日志、数据库慢查询、Redis 命中率、CPU/内存/网络和连接池。定位时要区分客户端压测机瓶颈和服务端瓶颈。

### Q4：报告页应该展示哪些内容？

**建议回答：**  
接口/UI 自动化报告看通过率、失败用例、失败原因、请求响应、截图/Trace、执行耗时和历史趋势。性能报告看 QPS、响应时间分布、P95/P99、错误率、资源指标和慢接口列表。报告要能帮助定位问题，而不是只展示一个通过率。

## 游戏测试迁移专项

### Q1：你没有游戏项目，为什么适合游戏测试？

**建议回答：**  
我不会把自己包装成有游戏项目经验，但我的测试开发能力可以迁移。游戏也有大量接口、状态、资产和性能问题，例如账号、背包、商城、抽卡、活动、战斗结算和排行榜。我熟悉接口自动化、UI 自动化、性能测试、日志报告和平台化执行，这些能帮助游戏 QA 做工具化和自动化。短板是引擎和游戏专项，我正在补 Unity/Unreal 测试、弱网、兼容性、帧率和崩溃定位。

**可能追问：** 你怎么快速上手一个游戏项目？

**追问回答方向：**  
先理解核心玩法和版本目标，再拆模块和风险点；搭建设备和环境，熟悉日志、包体、资源版本和后台工具；优先覆盖登录、账号、资产、支付、主线玩法和活动；同时补充弱网、兼容、性能和崩溃用例。

### Q2：怎么测试游戏背包系统？

**建议回答：**  
从功能、状态、边界和数据一致性四层测试。功能上看获得、使用、删除、合成、分解、排序、筛选；边界看背包满、数量为 0、数量上限、道具过期；状态看任务奖励、商城购买、战斗掉落、邮件领取是否都能正确入包；一致性看客户端展示、服务端数据和日志是否一致，断线重连后资产是否正确。

**可能追问：** 背包道具重复扣除怎么办？

**追问回答方向：**  
要查操作日志、接口请求、服务端幂等、客户端重复点击和网络重试。支付或资产类接口要有幂等 ID，防止重复提交导致重复扣除。

### Q3：怎么测试抽卡概率和保底？

**建议回答：**  
功能上验证单抽、十连、不同卡池、活动切换和结果展示；规则上验证概率配置、保底计数、重复角色转化、限定角色和卡池继承；数据上需要大量模拟或后台日志验证概率分布，但测试环境不应只靠手动抽。保底要重点测临界次数，例如保底前一次、保底触发、触发后计数重置。

**可能追问：** 概率问题怎么判断是不是 bug？

**追问回答方向：**  
单次异常不能说明概率 bug，需要看配置、随机种子、日志和足够样本统计。保底和计数类规则则可以通过确定性测试验证。

### Q4：弱网测试怎么做？

**建议回答：**  
弱网测试要模拟高延迟、丢包、抖动、断网、网络切换、切后台、锁屏等场景。重点看登录、战斗、匹配、支付、资源下载和断线重连。提交缺陷时要记录网络参数、操作时间、日志、录屏和账号状态。

**可能追问：** 断线重连后最关注什么？

**追问回答方向：**  
关注状态一致性。比如战斗是否能恢复、奖励是否重复发放、背包资产是否一致、匹配状态是否清理、支付是否补单、任务进度是否正确。
