# 7 天短期冲刺复习

## 使用原则

这份计划适合面试前一周使用。每天分成四块：

- 知识复习：当天必须重新过一遍的概念。
- 实战练习：当天要写或整理的材料。
- 简历输出：把知识点绑定到你的简历项目。
- 自测问题：晚上模拟面试。

每天建议 6-8 小时。如果时间只有 3-4 小时，优先级是：简历项目深挖 > 接口/Pytest > 性能与工程基础 > 游戏测试专项 > 手撕代码。

## Day 1：测试理论、用例设计与缺陷报告

### 知识复习

- 测试流程：需求评审、测试计划、用例设计、执行、缺陷、回归、上线验收。
- 用例方法：等价类、边界值、场景法、判定表、状态迁移、正交实验。
- 测试类型：功能、接口、UI、性能、兼容、安全、稳定性、探索式。
- 缺陷生命周期：提交、分配、修复、回归、关闭、重开。
- 冒烟与回归：一个看主链路可用，一个看变更影响面。

### 实战练习

选择 2 个题目写测试点：

- 微信红包。
- 电商购物车。
- 游戏登录和选服。
- 游戏背包道具使用。
- 游戏抽卡保底机制。

每个题目至少覆盖：

- 正常路径。
- 异常输入。
- 权限/状态。
- 边界值。
- 并发或重复操作。
- 日志和可观测性。

### 简历输出

把“测试开发基础”技能行变成面试口径：

```text
我比较熟悉从需求分析到回归验收的完整测试流程。用例设计上不会只覆盖正常路径，会结合等价类、边界值、场景法和状态迁移。比如在接口自动化框架里，我会对订单类接口覆盖正常下单、参数缺失、非法状态、重复提交、权限越权和数据库状态校验。
```

### 自测问题

- 怎么测试一个登录功能？
- 怎么测试一个游戏抽卡系统？
- 严重级别和优先级有什么区别？
- 冒烟测试和回归测试怎么划分？
- 如果开发说“这个不是 bug”，你怎么办？

## Day 2：接口测试、HTTP 与 Pytest 框架

### 知识复习

- HTTP 方法、状态码、请求头、Cookie/Session/Token。
- REST 接口测试点：参数、鉴权、权限、幂等、响应 Schema、错误码、数据库副作用。
- Pytest：Fixture、parametrize、mark、conftest.py、插件、hook。
- Requests：Session、统一请求封装、超时、重试、日志。
- Allure/JUnit XML：报告结构、失败截图/日志、CI 集成。

### 实战练习

整理一个接口自动化框架目录：

```text
config/
clients/
keywords/
cases/
data/
assertions/
reports/
utils/
```

写出每层职责，并准备 1 个示例：

- 登录接口如何封装。
- Token 过期如何刷新。
- 数据库断言如何做。
- 失败接口如何输出 curl。

### 简历输出

准备“接口自动化测试框架”3 分钟讲述：

```text
这个项目的目标是面向登录、用户、权限、商品、订单等接口搭建可复用自动化框架。我按配置层、API 客户端层、业务关键字层、用例层和报告层拆分。客户端层统一处理域名、请求日志、Token 刷新和超时；业务关键字层封装登录、创建订单等动作；用例层通过 Pytest 参数化和 mark 支持冒烟、回归、按模块执行。断言上除了响应字段，还会做 JSON Schema 和数据库状态校验，最后通过 Allure/JUnit XML 接入 CI。
```

### 自测问题

- Fixture 和普通函数封装有什么区别？
- `scope=session` 和 `scope=function` 适合什么场景？
- Token 自动刷新怎么避免多个用例重复登录？
- 接口返回 200 但业务失败，怎么断言？
- 如何避免测试数据互相污染？

## Day 3：Web UI 自动化与 Playwright

### 知识复习

- 定位器：role、text、label、CSS、XPath、test id。
- 等待：自动等待、显式等待、网络等待、元素可见/可点击。
- PO 模式：Page Object、业务动作封装、断言分离。
- 稳定性：截图、Trace、视频、失败重试、测试数据隔离。
- 常见场景：登录、搜索、表单、弹窗、文件上传、分页、筛选。

### 实战练习

选择一个简单页面或本地 demo，写出 PO 结构：

```text
pages/login_page.py
pages/order_page.py
tests/test_login.py
tests/test_order_flow.py
```

至少说明：

- 为什么选择语义化定位器。
- 为什么避免 `time.sleep()`。
- 失败时 Trace 如何帮助定位。

### 简历输出

准备“UI 自动化”回答：

```text
我在持续测试平台里用 Playwright 落地 PO 模式。我的思路是把页面元素和业务动作封装到 Page 层，用例只描述业务流程和断言。稳定性上优先使用 role、label、text 或 test id 这类语义化定位器，减少动态 class 和绝对 XPath；失败时保留 Trace、截图和视频，用于回放 DOM 变化、网络异常和终端报错。
```

### 自测问题

- Selenium 和 Playwright 的差异是什么？
- UI 自动化脚本为什么容易不稳定？
- PO 模式解决了什么问题？
- 什么时候应该做 UI 自动化，什么时候不应该？
- 游戏 UI 自动化为什么比 Web UI 更难？

## Day 4：性能测试、稳定性测试与瓶颈定位

### 知识复习

- 指标：并发用户数、QPS/TPS、平均响应时间、TP95/TP99、错误率、吞吐、资源利用率。
- 压测模型：阶梯加压、稳定性运行、峰值压力、容量评估。
- JMeter：线程组、取样器、断言、监听器、参数化、关联、命令行执行。
- Locust：用户行为建模、任务权重、分布式压测、Python 脚本化。
- 瓶颈：慢 SQL、连接池不足、缓存未命中、锁竞争、接口超时、资源耗尽。

### 简历输出

准备“性能测试”回答：

```text
我做性能测试时会先明确目标，是容量评估、稳定性验证还是瓶颈定位。比如登录、商品查询、订单提交这类场景，我会准备账号池和商品数据，用 JMeter 或 Locust 设计阶梯加压和稳定运行，统计 QPS、平均响应时间、P95、错误率和慢接口列表。定位时会结合服务日志、数据库慢查询和 Redis 命中情况，而不是只看压测工具报告。
```

游戏专项迁移：

- 服务端：登录洪峰、匹配队列、战斗结算、排行榜刷新。
- 客户端：FPS、帧时间、卡顿、内存、发热、耗电、资源加载。

## Day 5：CS 基础、Linux、Docker、MySQL、Redis、CI

### 知识复习

- Linux：进程、端口、日志、磁盘、内存、网络连接。
- Docker：镜像、容器、Dockerfile、Compose、volume、network。
- MySQL：索引、事务、隔离级别、MVCC、Explain、慢查询。
- Redis：数据结构、过期策略、缓存问题、队列、分布式锁。
- 网络：HTTP/HTTPS、TCP 三次握手四次挥手、状态码、长连接。
- CI：流水线触发、测试执行、报告归档、失败通知、质量门禁。

### 实战练习

准备这些命令的用途：

```bash

tail
grep

```

准备 3 个工程问题：

- 接口超时如何排查？
- CI 上失败但本地通过怎么办？
- Redis 队列任务丢失或重复执行怎么办？

### 简历输出

准备“测试平台工程落地”回答：

```text
持续测试平台里，我会把 Pytest、Playwright 和 Locust 的执行入口封装为平台任务，通过 Redis 队列异步调度并回收结果。平台需要维护任务状态、执行日志、报告地址、失败原因和历史趋势。工程上要处理并发限制、任务超时、失败重试、环境隔离和报告归档，这些比单纯在本地跑脚本更接近真实测试开发工作。
```

### 自测问题

- MySQL 索引为什么能加速查询？
- Redis 缓存穿透、击穿、雪崩怎么处理？
- Docker volume 和端口映射有什么用？
- CI 中如何选择冒烟集和回归集？
- Redis 队列如何保证任务状态可追踪？

## Day 6：游戏测试专项与 AI 应用测试

### 知识复习：游戏测试

- 模块：登录、选服、角色、任务、背包、装备、商城、抽卡、邮件、好友、匹配、战斗、排行榜。
- 专项：兼容性、弱网、断线重连、热更新、资源加载、崩溃、卡顿、帧率、内存、反作弊、本地化。
- 定位：设备型号、系统版本、游戏版本、资源版本、服务器、账号、网络、日志、录屏。
- 工具意识：Android logcat、Profiler、抓包、崩溃日志、性能监控。

### 知识复习：AI 应用测试

- Prompt/RAG 流程测试。
- Golden Dataset 和 Bad Case 回流。
- JSON Schema、关键词、相似度、规则断言。
- 多轮对话上下文保持、槽位更新、格式漂移、幻觉/拒答。

### 实战练习

写两个用例集：

1. 游戏抽卡系统测试点。
2. AI 客服多轮对话测试点。

每个用例集至少覆盖：

- 正常链路。
- 异常输入。
- 状态迁移。
- 数据一致性。
- 日志与证据。
