## 公开面经共性结论

### 结论一：测试开发不是“功能测试 + 会点 Python”

面试常见考察：

- 测试理论和用例设计。
- HTTP、接口测试、数据库校验。
- Pytest/Requests 自动化框架设计。
- UI 自动化稳定性。
- 性能测试指标和瓶颈定位。
- Linux、MySQL、Redis、Docker、CI。
- 简单手撕代码。

你的策略：

- 讲项目时避免只说工具名，要讲分层、数据、断言、报告、CI 和失败定位。
- 每个框架设计都要能画出目录结构和执行流程。
- 代码题不追求难题，先保证常见简单/中等题能稳定写对。

### 结论二：项目深挖会追问“你到底做了什么”

常见追问：

- 哪些模块是你负责的？
- 框架为什么这样分层？
- 数据怎么准备和清理？
- 自动化失败怎么定位？
- CI 跑哪些用例？
- 性能瓶颈怎么查？
- 指标怎么统计？

你的策略：

- 每个项目准备“职责边界、关键设计、失败案例、改进方向”。
- 遇到团队项目，不要说“我们做了”，要明确“我负责了哪些模块”。
- 不确定的数据不要硬编，用“当时没有量化，如果现在做我会这样统计”更稳。

### 结论三：游戏测试重视兴趣，但更重视测试思维

游戏测试面试常见考察：

- 是否理解游戏模块和玩家链路。
- 是否能设计完整测试点。
- 是否能定位崩溃、卡顿、弱网、兼容问题。
- 是否理解账号资产、支付、抽卡、活动、战斗结算的风险。
- 是否有游戏体验和缺陷敏感度。

你的策略：

- 不要只说“我喜欢玩游戏”。要说你如何从测试视角拆玩法、风险、状态和数据一致性。
- 准备 3 个游戏模块测试点：背包、抽卡、断线重连。
- 主动承认没有正式游戏项目，但强调测试开发能力可迁移，并说明你已补哪些专项。

### 结论四：性能测试会被问指标口径和定位链路

常见追问：

- 并发数和 QPS 的关系。
- 平均响应时间和 TP95 的区别。
- 压测数据怎么准备。
- JMeter 和 Locust 怎么选。
- 接口慢了如何定位。
- 压测报告怎么写。

你的策略：

- 先定义指标，再讲工具。
- 讲定位时按“压测报告 -> 服务日志 -> 数据库 -> 缓存 -> 系统资源”展开。
- 游戏性能要补客户端指标：FPS、帧时间、内存、发热、耗电、资源加载。

### 结论五：AI 应用测试是你的差异化，但也容易被质疑

常见追问：

- AI 回复没有标准答案，怎么测？
- Golden Dataset 怎么构建？
- Bad Case 如何分类？
- JSON Schema 能保证正确吗？
- Prompt 改了怎么做回归？
- RAG 召回错误怎么定位？

你的策略：

- 把 AI 测试讲成“传统自动化 + 非确定性评估”。
- 主动说明自动化断言的边界：Schema 只能保格式，事实正确还要规则、相似度、人工验收。
- 准备 3 类 Bad Case：上下文丢失、知识召回错误、格式漂移。

## 面试流程打法

### 开场自我介绍

目标：让面试官知道你是测试开发主线，兼具 AI 应用测试和游戏测试迁移潜力。

测试开发版：

```text
我目前是重庆邮电大学计算机技术硕士，求职方向是测试开发。我的经历主要集中在接口自动化、UI 自动化、性能测试和测试平台建设。实习中我做过大模型应用交付验证，负责模型服务接口联调、Prompt/RAG 流程测试和 Bad Case 分析。项目方面，我搭建过 Pytest + Requests 的接口自动化框架，也做过基于 FastAPI、Redis、Playwright 和 Locust 的持续测试平台。我希望做偏工程化、自动化和质量平台方向的测试开发。
```

游戏测试版：

```text
我目前投递测试开发和游戏测试方向。我的核心能力是自动化测试、接口测试、性能测试和测试平台建设，虽然没有正式游戏项目，但这些能力可以迁移到游戏账号、背包、商城、抽卡、匹配、战斗结算和活动测试。我也在补游戏专项，包括弱网、兼容性、帧率卡顿、崩溃日志、热更新和 Unity/Unreal 测试基础，希望后续在游戏 QA 中做更偏技术化和工具化的工作。
```

注意：

- 第一句明确岗位。
- 不要一开始堆所有技术名。
- 30 秒内抛出最强项目，让面试官顺着问。

### 项目讲述打法

使用“Q-P-A-R-B”结构：

- Quality Risk：这个项目要解决什么质量风险。
- Problem：原来的痛点是什么。
- Action：你做了什么设计和实现。
- Result：有什么结果或产出。
- Bad Case：遇到什么失败案例，怎么修。

示例：

```text
接口自动化框架的质量风险是接口数量增加后，手工回归成本高，而且接口变更容易漏测。
我把框架分成配置层、API 客户端层、业务关键字层、用例层和报告层。
客户端层统一处理请求、Token、日志和异常，用例层通过 Pytest 参数化和 mark 支持冒烟、回归和按模块执行。
断言上不仅看响应码，还做 JSON Schema 和数据库状态校验。
最后通过 Allure/JUnit XML 接入 CI，失败时可以看到请求参数、响应体、日志和可复现 curl。
```

### 回答追问打法

遇到追问时按这个顺序：

1. 先给直接结论。
2. 解释原因或原理。
3. 结合项目实践。
4. 补充局限和改进方向。

示例：

```text
我不会把失败重试作为默认策略，因为它可能掩盖真实缺陷。
在自动化框架里，我会限制重试场景，比如网络抖动、Token 过期或 UI 等待不稳定。
如果是业务断言失败、数据库状态错误或响应 Schema 不一致，就不应该重试到通过，而是保留日志和请求响应用于定位。
```

### 不会的问题怎么处理

可以这样说：

```text
这个点我没有在项目里完整实践过，但我理解它的基本思路是……
如果让我落地，我会先从……验证，再逐步补……
```

不要这样说：

- “这个我不会。”
- “应该是工具自动处理的。”
- “网上一般都是这么做。”
- “我们项目里不是我负责。”

更稳的替代：

```text
这个模块不是我主负责，但我了解它和我负责模块的接口边界。我能讲清楚输入输出和问题定位方式。
```

### 指标被质疑时怎么处理

稳妥回答：

```text
这个指标是项目内部口径，我会先说明统计方式和局限。
比如 100+ 条对话回归样例指的是我们沉淀的核心多轮对话和 Bad Case 样例，不代表覆盖所有真实用户问题。
如果要更严格评估，我会按场景分层，记录样本量、通过率、失败类型和版本差异。
```

核心原则：

- 指标越具体越可信。
- 主动承认局限比硬撑更专业。
- 区分工具报告、离线测试集和线上业务指标。

## 一面、二面、HR 面准备重点

### 一面

重点：

- 测试理论。
- 接口测试和 Pytest。
- CS 基础。
- 简单手撕代码。
- 简历初步追问。

准备：

- 每天 1-2 道代码题。
- 登录、订单、背包、抽卡等测试点题目。
- 接口自动化框架 3 分钟讲述。

### 二面

重点：

- 项目深挖。
- 框架设计和平台架构。
- 性能测试和瓶颈定位。
- 游戏专项或 AI 测试差异化。

准备：

- 每个项目画一张流程图。
- 每个项目准备 3 个 Bad Case。
- 每个指标准备口径。
- 准备“如果重做会怎么优化”。

### HR 面

重点：

- 求职动机。
- 岗位理解。
- 稳定性。
- 沟通协作。
- 对游戏行业或测试开发方向的理解。

准备：

- 为什么做测试开发。
- 为什么也考虑游戏测试。
- 研究生期间如何安排实习和毕业。
- 遇到开发不认 bug 如何沟通。
- 对加班、版本压力和线上问题的看法。

## 反问问题库

技术面可以问：

- 团队测试开发和业务测试的职责边界是怎样的？
- 当前自动化测试覆盖接口、UI、性能还是客户端专项？
- 团队是否有统一测试平台、用例管理和报告归档？
- 游戏项目中弱网、兼容性、性能和崩溃问题通常怎么定位？
- 新人入组后通常先负责业务测试、自动化工具还是专项测试？

HR 面可以问：

- 这个岗位对应届生或实习生的培养路径是怎样的？
- 团队对测试开发能力和业务测试能力的权重如何？
- 后续面试流程和时间安排大概是怎样的？

避免反问：

- 能不能不加班？
- 多久涨薪？
- 我这个表现能过吗？
- 游戏测试是不是比较简单？

- 强安全逆向/反作弊岗位。
- 纯服务端开发岗位。
