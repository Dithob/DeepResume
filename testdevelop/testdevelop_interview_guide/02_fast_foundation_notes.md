# 快速补基础知识点

## 使用原则

这份文档适合短期补基础和面试前速查。每个知识点都按“主要内容 + 面试要说清 + 简历关联”整理。优先级如下：

1. 先补测试开发主线：测试理论、接口测试、Pytest、UI 自动化、性能测试、CI 和工程基础。
2. 再补游戏测试专项：玩法模块、弱网、兼容性、性能、崩溃日志、Unity/Unreal 基础。
3. 最后补 AI 应用测试：Golden Dataset、Bad Case、RAG/Prompt 测试和结构化输出校验。

## 测试基础

### 测试流程

主要内容：

- 需求分析：理解业务规则、异常分支、数据状态和风险点。
- 测试计划：范围、资源、时间、环境、风险、准入准出标准。
- 用例设计：功能、接口、异常、兼容、性能、稳定性。
- 执行与缺陷：记录问题、提交缺陷、推动修复、回归验证。
- 总结复盘：覆盖率、缺陷分布、风险遗留、上线建议。

面试要说清：

- 测试不是最后才介入。需求评审阶段就应发现逻辑歧义和不可测问题。
- 准入标准是“什么时候开始测”，准出标准是“什么时候可以上线或交付”。

简历关联：

- 你的实习写了功能冒烟、回归用例设计、接口异常排查和线上 Bad Case 分析。需要能讲完整流程。

### 用例设计方法

主要内容：

- 等价类：把输入划分为有效/无效集合，每类选代表值。
- 边界值：重点测最小值、最大值、临界值、空值、超长值。
- 场景法：覆盖用户真实业务链路，例如登录-查询-下单-支付-退款。
- 判定表：多条件组合导致不同结果时使用。
- 状态迁移：对象状态变化明显时使用，例如订单、任务、角色状态。
- 正交实验：组合很多但无法穷举时，用较少组合覆盖主要因素。

面试要说清：

- 功能测试题不要只列正常路径，要覆盖异常输入、权限、状态、重复操作、并发和兼容。
- 游戏测试常用状态迁移，例如任务从未接取、已接取、可完成、已完成、已领取奖励。

### 缺陷报告

主要内容：

- 标题要准确：模块 + 问题 + 条件。
- 复现步骤要可执行：前置条件、数据、操作路径、实际结果、期望结果。
- 附件要完整：日志、截图、视频、接口请求响应、设备型号、版本号。
- 严重级别与优先级分开：崩溃通常严重级别高，但是否优先修取决于版本计划。

面试要说清：

- 好缺陷报告的目标是降低开发复现成本。
- 游戏缺陷必须写清设备、系统、包体版本、资源版本、网络环境和复现概率。

## 接口测试

### HTTP / REST 基础

主要内容：

- 方法：GET、POST、PUT、PATCH、DELETE。
- 状态码：2xx 成功、3xx 重定向、4xx 客户端错误、5xx 服务端错误。
- 鉴权：Cookie、Session、Token、JWT、OAuth2。
- 请求体：JSON、form-data、x-www-form-urlencoded、文件上传。
- 幂等性：重复请求结果是否一致，常见于订单提交、支付回调、领取奖励。

面试要说清：

- 接口测试不只看 200，还要检查业务错误码、响应字段、数据库状态和副作用。
- 权限测试要覆盖未登录、过期 Token、越权访问、角色权限和数据权限。

简历关联：

- 你的接口自动化框架面向登录、用户、权限、商品、订单，面试官很可能从鉴权和幂等切入。

### 接口测试点清单

必须覆盖：

- 正常参数、必填缺失、类型错误、边界值、非法枚举、特殊字符、超长文本。
- 鉴权、权限、Token 过期、重复提交、并发请求、幂等性。
- 响应 Schema、字段类型、业务错误码、排序分页、空数据。
- 数据库校验：新增、更新、删除、状态流转、事务回滚。
- 日志与可观测：请求 ID、错误日志、耗时、trace。

游戏接口补充：

- 登录、选服、角色创建、背包变化、商城购买、抽卡结果、战斗结算、邮件领取、排行榜刷新。
- 关注资产一致性：道具、货币、装备、经验、体力、活动次数。

## Pytest 与接口自动化框架

### Pytest 核心

主要内容：

- Fixture：管理前置/后置、测试数据、登录态、数据库连接。
- scope：function、class、module、session。
- parametrize：数据驱动，覆盖多组输入。
- mark：分类执行，例如 smoke、regression、api、slow。
- conftest.py：共享 Fixture 和 hook。
- 插件：pytest-html、pytest-xdist、pytest-rerunfailures、allure-pytest。

面试要说清：

- Fixture 的价值是隔离环境准备和测试逻辑，减少重复代码。
- 参数化不是为了炫技，而是让同一测试逻辑覆盖多组数据。
- 失败重试要谨慎，不能掩盖真实缺陷；适合处理网络抖动和 UI 偶发等待问题。

简历关联：

- 你写了公共断言、Fixture、参数化数据驱动、失败重试、日志采集和报告，需要准备代码结构图。

### 框架分层

推荐结构：

```text
config/       环境配置、域名、账号、数据库连接
clients/      API 客户端、统一请求封装、Token 刷新
keywords/     业务关键字，例如登录、创建订单、查询商品
cases/        测试用例
data/         YAML/JSON/CSV 测试数据
assertions/   公共断言、Schema 校验、数据库校验
reports/      Allure/JUnit XML 报告
utils/        日志、时间、随机数据、加密签名
```

面试要说清：

- 客户端层负责“怎么请求”，业务关键字层负责“业务动作”，用例层负责“验证什么”。
- 框架要支持环境切换、用例筛选、失败定位、报告展示和 CI 执行。

## Web UI 自动化

### 稳定脚本原则

主要内容：

- 优先使用 role、text、label、test id 等语义化定位器。
- 避免依赖动态 class、绝对 XPath 和固定 sleep。
- 用自动等待或显式等待处理元素可见、可点击、网络完成。
- 页面对象模式中，页面元素和业务动作封装在 Page 类，测试用例只表达业务流程和断言。
- 失败时保留截图、Trace、视频、浏览器控制台日志。

面试要说清：

- UI 自动化最大难点是稳定性和维护成本。
- Trace 不只是附件，它能定位 DOM 变化、网络请求失败和页面跳转问题。

简历关联：

- 你写了 Playwright PO 模式、智能等待、语义化定位器、Trace、失败截图与视频录制。

## 性能测试

### 指标口径

主要内容：

- 并发用户数：同时发起业务操作的虚拟用户数量。
- QPS/TPS：单位时间请求数或事务数。
- 平均响应时间：容易被极端值影响，必须结合 TP95/TP99。
- TP95/TP99：95% 或 99% 请求在该时间内完成。
- 错误率：失败请求占比，包括超时、5xx、断言失败。
- 资源指标：CPU、内存、磁盘 IO、网络 IO、数据库连接、Redis 命中率。

面试要说清：

- 并发数不是 QPS，用户思考时间、请求链路和响应时间都会影响吞吐。
- 性能测试要先明确目标：容量评估、瓶颈定位、稳定性验证还是压垮系统。

### 压测分析流程

推荐流程：

1. 明确场景：登录、查询、下单、提交订单。
2. 准备数据：账号池、商品池、订单参数、清理脚本。
3. 设计模型：阶梯加压、稳定运行、峰值压力。
4. 执行监控：接口指标、服务资源、数据库、缓存、日志。
5. 定位瓶颈：慢 SQL、连接池、缓存未命中、锁竞争、接口超时。
6. 输出报告：结论、瓶颈、风险、优化建议和复测结果。

游戏性能补充：

- 客户端看 FPS、帧时间、卡顿、内存、CPU/GPU、发热、耗电、包体、资源加载时间。
- 服务端看登录洪峰、匹配队列、战斗结算、排行榜刷新、活动开启高峰。

## 工程基础

### MySQL

必须掌握：

- 索引、联合索引、最左前缀、覆盖索引、回表。
- 事务 ACID、隔离级别、MVCC、锁。
- Explain、慢查询、分页优化。
- 测试中的数据库断言和数据准备/清理。

面试要说清：

- 数据库校验不是直接改库，而是验证接口副作用和业务状态是否正确。
- 自动化测试需要隔离测试数据，避免互相污染。

### Redis

必须掌握：

- String、Hash、List、Set、ZSet。
- 过期策略、淘汰策略、缓存穿透/击穿/雪崩。
- 队列、分布式锁的基本思路和风险。

简历关联：

- 你写了 Redis 队列异步调度测试任务，面试官可能问任务状态、失败重试、结果回收和并发控制。

### Linux / Docker / CI

必须掌握：

- Linux：查看进程、端口、日志、磁盘、内存、网络连接。
- Docker：镜像、容器、Dockerfile、Compose、volume、network。
- CI：触发条件、环境准备、执行测试、生成报告、失败通知、质量门禁。

面试要说清：

- CI 不是简单“跑一下脚本”，而是要决定哪些用例在提交后跑、失败如何阻断、报告如何归档。

## 游戏测试专项

### 高频模块测试点

登录/账号：

- 正常登录、密码错误、Token 过期、重复登录、异地登录、游客转正式账号、封禁账号。

背包/资产：

- 道具增加、消耗、堆叠、过期、排序、容量上限、断线重连后资产一致性。

商城/支付：

- 商品展示、限购、折扣、余额不足、重复支付、支付回调、补单、退款。

抽卡：

- 概率配置、保底机制、重复角色转化、十连结果、活动池切换、日志可追溯。

战斗/结算：

- 伤害计算、技能冷却、Buff/Debuff、死亡、重连、胜负判定、奖励结算。

弱网/兼容：

- 高延迟、丢包、断网、切后台、锁屏、来电、不同分辨率、低端机、系统版本。

### 游戏缺陷定位

必须记录：

- 设备型号、系统版本、游戏包体版本、资源版本、账号、服务器、网络环境。
- 复现路径、复现概率、录屏、截图、客户端日志、服务端请求 ID。
- 崩溃问题要附 crash log，卡顿问题要附性能数据和场景信息。

面试要说清：

- 游戏测试更强调复现条件，因为很多问题和设备、网络、资源版本、账号状态有关。

## AI 应用测试

### 非确定性输出怎么测

主要内容：

- 不能只用精确字符串断言，要组合规则、关键词、JSON Schema、相似度、人工标注。
- 对结构化输出优先做 Schema 校验、字段类型和值域校验。
- 对知识问答要验证引用来源、拒答策略和事实一致性。
- 对多轮对话要验证上下文保持、槽位更新和冲突澄清。

简历关联：

- 你在小啡 GPT 项目里做了关键词、JSON Schema、相似度和规则断言校验，可以作为 AI 测试亮点。

### Golden Dataset 与 Bad Case

主要内容：

- Golden Dataset：稳定、可复现、有人工标准答案或验收规则的测试集。
- Bad Case：线上或测试中发现的失败样本，按问题类型归因并回流。
- 版本对比：同一测试集上比较不同模型、Prompt 或检索策略。

面试要说清：

- Golden Dataset 不是越大越好，关键是覆盖典型场景、异常场景和高风险场景。
- Bad Case 要分类：召回错误、上下文丢失、格式漂移、幻觉、拒答不当、接口异常。
