## 公开面经的共性结论

### 结论一：算法岗仍然重视手撕代码

即使岗位写了大模型、RAG 或 AI 应用，校招和实习面试通常仍会安排算法题。常见形式：

- 线上笔试：2-4 道算法题，难度从 LeetCode 中等到偏难。
- 一面手撕：1-2 道题，要求边写边解释。
- 二面追问：算法题减少，项目和系统设计增加。

你的策略：

- 不追求刷题数量无限大，优先把常见题型刷熟。
- 每题都要能讲复杂度、边界条件和优化思路。
- 面试时先复述题意，再说思路，不要一上来沉默写代码。

### 结论二：AI 应用开发面试更看重“闭环能力”

公开资料和实际招聘 JD 中，AI 应用开发不只看会不会调用模型，而是看完整闭环：

- 业务问题是否定义清楚。
- 数据如何清洗、切分、标注和回流。
- 检索、生成、工具调用如何评估。
- 服务如何部署、压测、监控。
- 失败案例如何定位和修复。

你的策略：

- 讲项目时突出“从需求到上线/验收”的链路。
- 每个项目准备至少 3 个 Bad Case。
- 对每个指标说明统计口径和局限性。

### 结论三：RAG 项目最容易被问“是不是套壳”

面试官会判断候选人是否只会用 Dify/LangChain 搭流程。常见追问：

- 文档怎么切分？
- 为什么用这个 embedding？
- 向量检索为什么召回不到？
- BM25 和 Dense 怎么融合？
- Rerank 放在哪里？延迟如何控制？
- 评估集怎么构建？
- 如何处理幻觉和引用错误？

你的策略：

- 主动讲“我做了哪些非工具默认能力”：元数据增强、业务规则过滤、SQL 自纠错、Golden Dataset、Bad Case 回流。
- 准备 RAG 白板图：数据侧、查询侧、检索侧、生成侧、评估侧。

### 结论四：Text-to-SQL 的重点是安全和正确性

Text-to-SQL 面试不会只问 Prompt，常见追问是：

- 复杂 Schema 如何召回？
- Join 怎么处理？
- 生成 SQL 如何校验？
- 如何防止危险 SQL？
- 执行成功和结果正确有什么区别？
- 业务口径不一致怎么办？

你的策略：

- 把 SQL 链路讲成“候选 Schema -> SQL -> 预执行 -> 自修正 -> 结果解释 -> 反馈回流”。
- 必须主动提安全：只读账号、白名单、超时、行数限制、敏感字段脱敏。

### 结论五：模型微调会被追问数据和评估

如果简历写了 Qwen 微调、LoRA、SFT，面试官很可能问：

- 数据从哪里来？
- 数据量多大？
- 怎么清洗？
- 如何划分训练集和验证集？
- LoRA 参数怎么选？
- 微调后如何评估？
- 为什么不只用 RAG 或 Prompt？

你的策略：

- 不要把微调说成万能知识注入。更稳的说法是：微调用于风格、格式、领域话术和任务习惯；事实知识仍依赖 RAG。
- 准备一个“数据处理流程”：脱敏、去重、过滤、格式化、划分、人工抽检。

### 结论六：工程问题会区分 AI 应用候选人的成熟度

AI 应用开发岗位经常问：

- 接口如何设计？
- 超时如何处理？
- 如何压测？
- 如何降级？
- 如何记录日志？
- 如何控制成本？
- 如何处理并发？

你的策略：

- FastAPI、Docker、vLLM、JMeter 不要只作为技能词出现，要能讲真实链路。
- 准备一个“线上问题定位故事”：例如延迟升高、召回为空、SQL 报错、模型输出格式错误。

## 面试流程打法

### 开场自我介绍

目标：让面试官知道你不是泛泛做 AI，而是有 RAG/Agent/Text-to-SQL 落地经验。

推荐结构：

```text
我目前是重庆邮电大学计算机技术硕士，求职方向是算法工程师和 AI 应用开发。
我的经历主要集中在 LLM 应用落地，包括 RAG、Text-to-SQL、多轮对话记忆、模型微调和推理部署。
实习中我参与了家电售后场景的小啄 GPT 智能回复模型，做过 Qwen 微调、Dense/BM25 混合检索、RRF 融合和 Rerank。
项目方面，我做过一个医药数据分析问答助手，面向 800 多张业务表构建 RAG + Text-to-SQL 链路，并通过 SQL 自纠错和 Golden Dataset 评估把执行成功率从约 70% 提升到 90%+。
我希望应聘的方向是能把大模型和实际业务系统结合起来的算法或 AI 应用开发岗位。
```

注意：

- 不要一开始讲太多工具名。
- 第一句就要明确目标岗位。
- 30 秒内抛出最强项目，让面试官顺着问。

### 项目讲述打法

使用“P-T-E-R-B”结构：

- Problem：业务问题和技术难点。
- Task：你负责什么。
- Engineering：具体方案和技术选择。
- Result：指标、效果、业务价值。
- Bad Case：失败案例和迭代。

示例：

```text
这个项目的难点不是简单把问题交给大模型，而是 800 多张表下的 Schema 选择和 SQL 正确性。
我负责核心 LLM 链路设计，先对数据字典做元数据增强，再通过向量召回和业务规则过滤选出候选表字段。
SQL 生成后我加入预执行和错误回传机制，让模型根据 MySQL Error 重写。
评估上构建了约 200 条 Golden Dataset，同时看执行成功率、RAGAS 相关指标和人工 Bad Case。
最后执行成功率从约 70% 提升到 90%+。失败案例主要集中在多表 Join、业务口径歧义和枚举值别名，后续通过样例 SQL 和字段别名增强继续优化。
```

### 回答追问打法

遇到追问时按这个顺序：

1. 先给直接结论。
2. 再说明原因或原理。
3. 再联系项目实践。
4. 最后补充局限和改进。

示例：

```text
RRF 的作用是融合不同召回器的排序结果，尤其适合 BM25 和 Dense 这种分数尺度不一致的情况。
如果直接加权，BM25 分数和向量相似度分数没有天然可比性，调参会比较不稳定。
所以我在客服知识检索里先让 BM25 和 Dense 各自召回，再按排名做 RRF 融合，最后对候选结果做 Rerank。
它的局限是没有利用绝对分数信息，所以后续可以结合学习排序或基于验证集调权重。
```

### 不会的问题怎么处理

可以这样说：

```text
这个点我没有在线上项目里完整实践过，但我理解它的基本思路是……
如果让我在项目中落地，我会先从……开始验证。
```

不要这样说：

- “这个我不会。”
- “我们项目里别人做的。”
- “Dify/LangChain 应该自动处理了。”
- “网上一般都是这么做的。”

### 指标被质疑时怎么处理

稳妥回答：

```text
这个指标是当时项目内部口径，不是公开 benchmark。
我会把它拆成三部分解释：样本集规模、baseline、统计方式。
比如执行成功率是指 SQL 能在 MySQL 上成功执行，不代表结果一定完全正确，所以我们还结合人工验收和 Bad Case 回流做补充评估。
```

核心原则：

- 指标越具体越可信。
- 主动承认局限，比硬撑更专业。
- 区分离线指标、线上指标和业务指标。

## 一面、二面、HR 面准备重点

### 一面

重点：

- 算法题。
- ML/DL/LLM 基础。
- 简历项目初步追问。

准备：

- 每天 2-3 道算法题。
- Transformer、RAG、LoRA、vLLM 高频题背熟。
- 自我介绍控制在 1-2 分钟。

### 二面

重点：

- 项目深挖。
- 架构设计。
- 指标和工程落地。
- 技术取舍。

准备：

- 每个项目画架构图。
- 每个项目准备 3 个 Bad Case。
- 每个指标准备统计口径。
- 准备“如果重做会怎么优化”。

### HR 面

重点：

- 求职动机。
- 稳定性。
- 沟通协作。
- 对岗位理解。
- 薪资和城市期望。

准备：

- 为什么选择 AI 应用开发。
- 为什么选择这家公司。
- 研究生期间如何安排实习和毕业。
- 项目中如何和业务、后端、测试协作。

## 反问问题库

技术面可以问：

- 团队当前的大模型应用主要是偏 RAG、Agent、模型训练还是业务平台建设？
- 线上 RAG 系统主要关注哪些指标？召回、生成、延迟和成本如何权衡？
- 团队是否有自己的评估集、Bad Case 回流和自动化评测平台？
- 新人入组后通常会负责哪类模块？

HR 面可以问：

- 这个岗位对应届生或实习生的培养路径是怎样的？
- 团队对算法工程和工程开发能力的权重如何？
- 后续面试流程和时间安排大概是怎样的？

避免反问：

- 能不能不加班？
- 多久涨薪？
- 我这个表现能过吗？
- 公司大模型业务是不是很水？

## 投递策略

优先投递 JD 中出现这些关键词的岗位：

- RAG
- Agent
- LLM Application
- 大模型应用
- AI 应用开发
- Text-to-SQL
- 智能客服
- 知识库问答
- 工具调用
- 模型服务部署
- LangChain / Dify / FastAPI / vLLM

谨慎投递只强调这些关键词的岗位：

- CV 检测/分割/跟踪。
- 推荐系统召回/排序。
- 大规模预训练。
- 强化学习算法研究。
- CUDA / 算子优化。

如果投传统算法岗，简历和面试需要额外强化：

- ML 基础。
- 深度学习论文和实验。
- 手撕代码。
- 传统 NLP/CV/推荐项目。
